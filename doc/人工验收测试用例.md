## 1. 测试环境准备

### 1.1 启动后端、Web、Mobile

终端 A — 启动后端 + Web：

```bash
cd /home/ranmaoyin/course-manage-system
npm run dev
```

终端 B — 启动 Mobile（Android 模拟器优先；iOS/Expo Go 亦可）：

```bash
npm run start --workspace @course/mobile
# 或
npm run android --workspace @course/mobile
```

终端 C — 验收前质量检查（推荐执行一次）：

```bash
npm run typecheck
npm run lint
npm run build
npm run test
```

### 1.2 接入地址

| 项目 | 地址 |
| --- | --- |
| API 健康检查 | `http://localhost:4100/health` |
| Web 端 | `http://localhost:5173/`（若被占用见 Vite 日志） |
| Web API Base URL | `http://localhost:4100/api/v1` |
| Mobile（Android 模拟器） API Base URL | `http://10.0.2.2:4100/api/v1` |

### 1.5 是否重置数据库

正式验收前如需干净状态：

```bash
npm run db:reset --workspace @course/server
npm run seed
```

`db:reset` 会按 `TABLE_NAMES` 列表 DROP+CREATE 所有表；`seed` 会写入种子教师/教务员。正式答辩临场不建议执行。

## 2. 用户信息管理（对应 需求 3.1.1）

### AC-USR-01 学生注册（Web）

- **关联需求**：3.1.1 ①用户注册  
- **角色**：学生甲  
- **测试端**：Web  
- **前置条件**：手机号 `13820260501` 尚未注册；后端在 `local` 验证码模式下（`config.allowVerificationPreview === true`）

| # | 操作 | 预期结果 |
| --- | --- | --- |
| 1 | 打开 `http://localhost:5173/`，登录页点击「学生注册」 | 进入注册表单，包含手机号、密码、确认密码、用户名、真实姓名、学号、验证码 |
| 2 | 输入 `13820260501 / Student123! / Student123! / student_jia / 学生甲 / S20260501` | 字段无报错 |
| 3 | 点击「获取验证码」 | 提示「验证码已发送」且 **页面回显 6 位预览验证码**（接口 `data.previewCode` 非空） |
| 4 | 验证码自动/手动填入后点击「完成注册」 | 弹出成功提示，自动返回登录页 |
| 5 | 用 `13820260501 / Student123!` 登录 | 成功进入学生工作台，顶部显示「学生甲」「学生」角色徽标 |

异常用例：
- 两次密码不一致 → 表单本地校验失败，提示「两次输入的密码不一致」
- 同一手机号再次注册 → 接口返回 `PHONE_ALREADY_REGISTERED`，提示「手机号已被注册」
- 重复使用同一学号 → 提示 `STUDENT_ID_ALREADY_REGISTERED`
- 验证码错误/过期 → 提示 `VERIFICATION_CODE_INVALID` / `VERIFICATION_CODE_EXPIRED`

### AC-NFR-01 响应时间观测（需求 3.2.2）

- **方法**：浏览器 DevTools「Network」/Mobile Expo Inspector，记录关键接口端到端耗时

| 接口 | 期望 | 验收方法 |
| --- | --- | --- |
| `POST /auth/login` | < 1 s | 登录按钮按下到接口完成 |
| `POST /auth/verification-code` | < 2 s | 获取验证码点击到 toast |
| `POST/PATCH/DELETE /courses` | < 1 s | 课程 CRUD |
| `POST /courses/:id/assignments`、`PATCH /assignments/:id`、`/cancel` | < 1–2 s | 作业 CRUD |

| `POST /submissions/:id/grade` | < 1 s | 批改 |
| `POST/PATCH/DELETE /feedbacks`、`/responses`、`/course-feedbacks` | < 1 s | 反馈/回答/课程反馈 |
| 任意 `GET *` 查询 | < 1.5 s | 列表查询 |

> 验收通过标准：连续 5 次操作的 95th 百分位满足。如超时，先检查后端日志 `logs/server-combined.log` 与数据库连接。

### AC-NFR-02 输入校验与精度（需求 3.2.1、3.3）

- 在每个表单依次故意输入：空字符串、超长字符串（>1000 字）、特殊字符 `<script>alert(1)</script>`、纯数字、SQL 关键字 `' OR 1=1--`  
- **预期**：
  - Zod 在前后端给出友好中文报错
  - 后端永远参数化绑定（不存在 SQL 注入风险）
  - 渲染时不执行任何 HTML/JS（XSS 防护）

### AC-NFR-03 故障恢复与数据完整性（需求 3.5）

### 14.3 评分项 ② 引入先进的信息处理技术 / 软件开发技术（10 分）

**自评目标档位：8–10。** 演示时着重讲以下技术选型与算法优化点：

- **全栈 TypeScript 严格模式**：`apps/server`、`apps/web`、`apps/mobile`、`packages/*` 共用 TS 5.x；CI 中 `npm run typecheck` 任一包错误即失败。
- **Zod schema 作为单一事实源**：`packages/shared/src/index.ts` 暴露 `loginSchema / studentRegisterSchema / courseCreateSchema / assignmentDraftSchema / submissionGradeSchema / feedbackSchema / courseFeedbackSchema / responseSchema / userStatusUpdateSchema` 等；**前后端共用**同一份 Zod，杜绝校验漂移。
- **三层鉴权**：JWT (`@fastify/jwt`) + Refresh Token（`nanoid(32)` + `bcrypt` 哈希入库 `auth_sessions`）+ 角色守卫 `requireRole`。改密、改号、注销时统一 `DELETE FROM auth_sessions WHERE user_id = ?`，强制所有端下线，避免会话漂移。
- **验证码双通道**：`config.verificationProvider`=`local`（演示/测试，6 位随机码 + 5 min TTL）/ `cloudbase`（实名 CloudBase 短信验证，`POST /auth/v1/verification`+`/verify`），通过同一份 `assertVerificationCode` 状态机切换，提供平滑回退策略。
- **数据访问层抽象**：`apps/server/src/lib/db/client.ts` 用 thin sync-shaped adapter 包裹 `mysql2/promise`，业务层只写 `await db.prepare(sql).get/all/run(...)`，**强制参数化绑定**杜绝 SQL 注入。
- **统一响应包络** `{ success, message, data, error, meta.requestId }`：所有接口经 `errorResponse / sendCreated`，错误经 `AppError → toAppError → setErrorHandler` 集中化，Web/Mobile 端的 `ApiError` 解包逻辑保持一致。
- **结构化日志**：`createLogWriter` 输出 NDJSON 到 `logs/server-combined.log` 与 `logs/server-error.log`，每条日志携带 `requestId`，便于跨端追踪。
- **算法优化亮点**（演示时可直接拿源码佐证）：
  - **查询 N+1 消除**：`apps/server/src/modules/assignments/routes.ts` 在学生作业列表里通过一次查询拿到所有 `submissions`，再用 `Map<assignmentId, submission>` 内存映射回填 `mySubmission`，避免逐条二次查询。
  - **课程列表 JOIN 一次成型**：`courses` 路由用 `LEFT JOIN users AS teachers` + `LEFT JOIN course_enrollments` 在单条 SQL 内带出教师姓名、学号以及当前学生是否已选课的 `enrolled` 标志位。
  - **校验码取最新**：`getLatestVerificationCode` 用 `ORDER BY created_at DESC LIMIT 1`，避免遍历历史；并按 `used_at` / `expires_at` / `code` 三段式校验，且 CloudBase 模式下额外调用远端 verify。
  - **幂等提交**：`POST /assignments/:id/submissions` 检测同学生已存在 submission 时改走 `UPDATE`，并阻断 `graded` 状态再次提交（`SUBMISSION_ALREADY_GRADED`）。
  - **注销账号 → 手机号/学号可复用**：`register/student` 检测到 `cancelled` 状态的同手机号/学号会 `UPDATE` 复用 `users.id`，而不是新增孤儿记录。
  - **取消作业的级联**：`POST /assignments/:id/cancel` 在一次事务内先 `DELETE FROM submissions WHERE assignment_id=?` 再把作业状态置为 `cancelled`，保证“取消→提交清空”的原子性。
  - **可编辑约束**：`assertAssignmentEditable` 把“截止时间过 → 不可改”和“已有提交 → 不可改”合并成一个守卫函数，业务路由直接调用，重用且无遗漏。
  - **响应缓存失效**：Mobile 端 `apps/mobile/src/query-invalidation.ts`（含单测）封装了 TanStack Query 的失效键策略，Web/Mobile 双端写操作后会精准 invalidate 关联 query，避免“写完不刷新”的视觉延迟。
- **包结构与依赖治理**：
  - `npm@10.9.4` workspaces 单仓多包，避免冗余依赖；
  - Vite 别名 `createRequire` 去重 React/ReactDOM，规避 monorepo hoist 双副本问题；
  - 服务端构建后 `scripts/write-dist-package-json.mjs` 注入 `{ "type": "commonjs" }`，源码 ESM、产物 CJS，避免 Node ESM/CJS 互导坑。
