# 课程互动管理系统假设与冲突处理

## A1. 技术栈升级假设

- 来源冲突：文档中出现了 `JSP + JavaWeb + Spring Boot + MySQL + Android Native` 的实现倾向，但宪章允许在不违背业务的前提下选择更现代、可维护的栈。
- 处理结论：实现采用 TypeScript monorepo（Web: React/Vite；Mobile: Expo React Native；Server: Fastify/TypeScript），通过共享类型、共享契约和统一 API 保持需求忠实。
- 原因：更适合在有限周期内同时交付 Web、移动端、共享契约、测试和可维护代码。
- 影响：界面与工程形式与原始 Java/JSP 不同，但业务流程、角色权限、数据模型与运行目标保持一致。

## A2. 数据库本地演示假设

- 来源冲突：文档指向 MySQL，但本地课程项目更需要零门槛演示。
- 处理结论：本地开发和测试默认使用 SQLite 文件数据库；仓库层和 SQL 结构保持关系型设计，可迁移到 MySQL。
- 原因：降低评审运行成本，保证仓库开箱即跑。
- 影响：`docs/DEPLOYMENT.md` 中同时说明本地演示模式和迁移到 MySQL 的路径。

## A3. 反馈模型拆分假设

- 来源澄清：需求获取报告同时包含“问题反馈发布模块”和独立“课程反馈模块”，两者面向的业务对象不同。
- 处理结论：围绕作业/批改结果的提问与反馈使用 `feedbacks` + `responses`；面向课程内容、教学方法、教师授课、学习收获等维度的课程反馈使用独立 `course_feedbacks`。
- 原因：避免把课程评价强行绑定到某一次作业提交，满足报告第 26-29 个用例。

## A4. 课程选择能力补充假设

- 来源缺口：文档多处要求学生查看“自己所选课程”的作业，但领域类图未显式给出选课/加入课程关系。
- 处理结论：补充 `course_enrollments` 关系，学生可加入课程，教务员可查看和管理。
- 原因：否则无法闭合“学生查看自己课程作业”的主流程。

## A5. 教师/教务员账号来源假设

- 来源冲突：文档写明“教师和教务员无需注册，由后台分配”，但找回密码、信息修改等流程对所有角色开放。
- 处理结论：教师和教务员账号由系统预置或由教务员创建，不开放自注册；登录、修改资料、重置密码依然支持。
- 原因：同时满足账号治理与后续使用流程。

## A6. 短信验证码实现假设

- 来源冲突：文档要求短信验证码，但项目环境未提供真实短信网关。
- 处理结论：验证码由后端生成并写入安全日志及开发态调试视图，生产接入真实短信服务的接口留在 `VerificationService` 抽象层。
- 原因：保证功能可演示、可测试，同时不伪造第三方能力。

## A7. 作业附件与答案附件范围假设

- 来源缺口：事务逻辑中提到了附件，但正式需求表以文本字段为主。
- 处理结论：第一版核心交付支持文本型作业说明与文本型答案；附件上传作为可扩展点在 API 和数据模型中预留。
- 原因：优先保证核心流程完整性与稳定性。

## A8. 同步策略假设

- 来源缺口：文档强调 Web/移动端一致，但未强制实时协议。
- 处理结论：采用“后端单一事实源 + 变更后主动失效刷新 + 定时拉取 + 手动刷新”的强一致演示策略。
- 原因：复杂度低、移动端兼容性好，能满足课程项目演示与同步要求。
