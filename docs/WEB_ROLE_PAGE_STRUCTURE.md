# Web 端角色页面结构设计

更新日期：2026-05-20

本文档记录 Web 端按角色组织的页面结构。覆盖公共认证、学生端、教师端、教务员端。

## 设计基线与约束

- 功能范围以 `doc/001 B 课程互动管理系统 需求获取报告 5.docx` 为准。
- 本次重构只调整 Web 前端的导航与页面组织方式，**不修改后端 API 与数据库 schema**。当前后端已完整覆盖需求获取文档的 25 个功能点，对应关系见末尾 §5「需求→后端→Web 入口」对齐表。
- 真实业务链路围绕教学上下文组织：

```text
课程 -> 作业 -> 提交/批改 -> 作业问题或反馈 -> 教师答疑回复
```

- 左侧导航只承载大的业务空间，具体操作进入「课程工作区」内完成；不再每个需求模块独占一个左侧入口。
- 任何超出需求获取文档但已经在后端实现的能力（典型如「课程整体反馈」），均显式标注为「设计增强」，并限定为与现有后端 schema 一致的最简形态，不在文档中再扩张。

## 0. 公共认证与账号生命周期

### 0.1 登录

支持学生、教师、教务员使用手机号 + 密码登录。

- 登录成功后按角色跳转到对应工作台：学生 → 学生工作台；教师 → 教师工作台；教务员 → 教务员工作台。
- 登录失败时必须明确区分：手机号或密码错误、账号已禁用、账号已注销；禁用或注销账号不能继续进入系统。

### 0.2 学生注册

学生注册页只面向学生开放。

流程：

- 输入手机号 → 获取并填写验证码 → 设置密码 → 二次确认密码 → 填写学生基础信息。

教师和教务员**不通过 Web 自助注册**。需求获取文档明确「教师和教务员无需注册，由账号由后台分配」，本系统中由系统管理员在初始化阶段通过 seed 预置（详见 §4.4）。

### 0.3 密码找回

支持学生、教师、教务员通过注册时绑定的手机号找回密码。

流程：

- 输入手机号 → 获取并填写验证码 → 设置新密码 → 二次确认新密码 → 重置成功后返回登录页。

### 0.4 账号维护

登录后，各角色都有「账号维护」入口承载账号生命周期相关功能。

按需求获取文档 3.1.1③，个人资料**只可修改手机号和密码**两项，账号维护页面只暴露这两项的编辑入口：

- 查看个人资料（只读展示姓名、学号/工号、学院、专业、班级等注册时录入的字段，不提供编辑）
- 修改手机号
- 修改密码
- 注销账号
- 退出登录

修改手机号需要旧手机号验证 + 新手机号验证两次验证码。修改密码需要旧密码 + 新密码 + 新密码确认。

注销账号属于危险操作，需要二次确认。注销后该账号不可继续登录；学生账号如需再次使用系统，应重新注册；教师/教务员账号注销或禁用后的恢复应走管理流程。

## 1. 总体导航原则

- 左侧导航只列入业务空间，不与需求获取文档的功能模块一一映射。
- 同一业务对象允许有「跨课程汇总入口」+「单课程深入入口」两条路径，但都指向同一详情页，避免割裂。
- 教师端区分两类入口：
  - **作业管理**：面向「作业对象」，关注作业的发布、修改、删除、统计。
  - **教学任务**：面向「待办任务」，关注教师当下需要批改或回复的事项。
- 教务员端不直接介入教师的作业批改和答疑流程，只读相关关联信息。

## 1.1 路由设计

新版采用「角色 + 业务空间 + 上下文 ID + 子 Tab」的层级 URL，便于刷新后保留上下文，也便于直接链接到具体作业 / 反馈线程。

约定：

- 所有受保护路由都以 `/<role>` 开头，`<role> ∈ {student, teacher, officer}`，由路由守卫校验当前登录角色与 URL 是否匹配，不匹配则重定向到该角色的 dashboard。
- 课程工作区使用 `/<role>/courses/:courseId/<tab>` 三段式；缺省 tab 重定向到 `overview`。
- 作业详情、反馈线程作为课程工作区内的子页面，URL 携带完整上下文：`/<role>/courses/:courseId/<tab>/:itemId`。
- 跨课程入口（学生「我的作业」、教师「作业管理 / 教学任务」、教务员「用户管理 / 课程反馈查看」）放在角色根下，不嵌套在课程下。

### 1.1.5 路由守卫与未匹配处理

- 路由守卫顺序：
  1. 检查是否已登录，未登录重定向到 `/login`。
  2. 检查 URL 中的 `<role>` 是否匹配当前用户角色，不匹配重定向到本角色的 `/<role>/dashboard`。
  3. 检查动态参数 `:courseId / :assignmentId / :feedbackId / :submissionId` 是否存在且当前用户有权限访问，否则展示 404 占位并提供回到上一级的按钮。
- 课程工作区刷新时，根据 URL 的 `:courseId` 重新拉取课程上下文，不依赖内存中的 `selectedCourseId`。
- 切换角色或注销登录后，必须清空 TanStack Query 缓存，防止跨账号串数据。

## 1.3 重构迁移建议步骤

按下面顺序推进，每完成一步可以独立运行 `npm run dev --workspace @course/web` 验证，避免一次性大改：

1. **路由骨架先行**：引入 React Router 嵌套路由（建议放 `apps/web/src/routes/`），按 §1.1 把所有路径写好；每个路由先指向占位组件 `<TODO route="..." />`。删除 `parseRouteView` / `viewPath` / `WorkspaceView` 联合类型。
2. **抽出 RoleShell**：把 `App.tsx` 内的左侧导航 + 顶部 header + 主内容区抽成 `RoleShell` 布局组件，根据 `<role>` 决定导航项；账号维护保持为子路由。
3. **学生端课程工作区**：实现 `<CourseWorkspace>` 容器（顶部课程上下文条 + Tab 切换 + Outlet），先迁移「课程概览」「作业」两个 Tab；作业详情走子路由 `assignments/:assignmentId`。
4. **学生端作业反馈与课程整体反馈 Tab**：把 `interaction` 与 `courseFeedbacks` 视图迁移成课程工作区内的 Tab，删除原左侧入口。在作业详情已批改区下方补一个「我有问题/反馈」按钮，对应 §2.4 / §2.5 主入口。
5. **教师端课程工作区**：复用容器，实现「课程概览/作业/提交批改/作业反馈/课程整体反馈」5 个 Tab；从课程外的左侧 `courses` 视图删掉教师端的作业内联表单。
6. **教师端跨课程入口**：实现 `/teacher/assignments`（作业管理对象流）与 `/teacher/tasks`（教学任务待办流），后者直接复用 `TeacherTaskWorkspace.tsx`，前者为新增。
7. **教务员端**：实现课程详情（4 个 Tab；删除/修改课程的确认弹窗中显式列出影响范围）+ 用户管理 3 Tab（去掉创建教师账号入口）+ 全局课程反馈查看。
8. **清理**：删除废弃的 `WorkspaceView` 字符串、过时的路由解析逻辑、不再使用的内联 JSX；让 `App.tsx` 行数减到只剩入口装配代码（路由表、QueryClient、ErrorBoundary、ThemeProvider 之类）。

## 2. 学生端

### 2.1 左侧导航

```text
工作台
我的课程
我的作业
账号维护
```

| 入口 | 职责 |
| --- | --- |
| 工作台 | 跨课程汇总当前最需要处理的学习待办：待提交作业、已批改作业提醒等；待办全部由 `submissions`、`feedbacks`、`responses` 当前 API 派生，不引入独立通知/消息表。 |
| 我的课程 | 查看可加入的课程与已加入的课程；进入某门课程后进入「课程工作区」。 |
| 我的作业 | 跨课程汇总所有作业状态。点击作业后进入对应课程工作区的「作业」Tab 的作业详情。即「我的作业」与「课程工作区→作业」是同一作业详情的两条切入路径，不是冗余。 |
| 账号维护 | 个人资料展示、修改手机号、修改密码、注销账号、退出登录。 |

### 2.2 课程工作区

学生从「我的课程」进入某门课程后进入课程工作区。

顶部固定展示课程上下文：

```text
课程名称 / 授课教师 / 学期 / 上课时间地点 / 课程状态 / 我的加入状态
```

Tab：

```text
课程概览 / 作业 / 作业反馈 / 课程整体反馈
```

> 「课程整体反馈」Tab 标注为「设计增强」（详见 §2.6）。

### 2.3 课程概览

只展示上下文，不承担密集操作。

- 课程简介
- 授课教师
- 学期
- 上课时间与地点
- 课程状态
- 课程人数 / 我的加入状态
- 最近作业摘要

### 2.4 作业

学生端主操作区。

作业列表区分两个维度，避免把课程作业本身状态与学生个人提交状态混在一起：

- **作业状态**（来自 `assignments.status` + 时间）：未开始 / 进行中 / 已截止 / 已取消
- **我的提交状态**（来自 `submissions.status`）：未提交 / 已提交 / 已批改

进入某次作业详情后，在同一页面完成：

- 查看作业说明、要求、开始时间、截止时间。
- 提交答案。
- 在「作业未截止且我的答案未批改」条件下修改答案。
- 查看分数和教师批改回复。
- **作业详情中，若我的提交已被批改，提交结果区下方提供「我有问题/反馈」按钮，直接打开反馈编辑器**——这是学生发起作业反馈的主入口。

作业详情页天然带有以下上下文：

```text
courseId / assignmentId / submissionId
```

> 与需求一致性说明：需求获取文档 3.1.4②「修改答案」描述为「学生在截止日期或教师批改前，可以修改自己之前的答案」。后端实现遵循该规则；UI 在不满足条件时应禁用「修改答案」按钮并显示原因。
>
> 关于「删除答案」：需求获取文档 3.1.4 列出的子功能为「提交答案 / 修改答案 / 批改答案 / 查询答案」，未列出「删除答案」。本设计不在 Web 端提供答案删除入口；如需放弃作业，学生可在允许窗口内将答案清空后重新提交。
>
> 关于「作业取消」：需求获取文档 3.1.3③ 明确「删除作业」时「系统将该作业的状态改为已取消、清除所有学生对该作业的提交记录」。后端 `POST /assignments/:id/cancel` 已实现该语义（状态改为 `cancelled` 并物理删除该作业下所有 submissions）。Web 端对学生展示时，作业状态变为「已取消」并显示取消原因，学生在该作业下的提交记录不再可见。

### 2.5 作业反馈

作业反馈不再作为左侧独立主入口，而是课程工作区内的上下文功能。

**入口有两条**：

1. **主入口**：作业详情 → 已批改区下方「我有问题/反馈」按钮。
2. **辅助入口**：课程工作区「作业反馈」Tab，按「已批改作业」列出可反馈对象与已发起的反馈线程，方便跨作业回看。

辅助入口列表每条记录展示：

- 作业名称
- 我的分数
- 教师批改回复摘要
- 是否已发起问题或反馈
- 教师是否已回复

进入反馈线程后展示：

- 学生提交答案（只读）
- 教师批改分数与批改回复（只读）
- 学生问题或反馈内容
- 教师回复列表（按时间排列）
- 当前处理状态（由是否有教师回复自然推导：未回答 / 已回答）

学生在反馈线程中可以：

- 查看自己发布的问题或反馈
- 修改自己发布的问题或反馈
- 删除自己发布的问题或反馈

发起条件（与后端 `POST /submissions/:submissionId/feedbacks` 一致）：

- 学生必须是该提交的所有者。
- 该提交必须已被教师批改（`submissions.status = 'graded'`）。

修改或删除问题/反馈前由后端校验记录归属。若教师已经回答，仍允许学生修改或删除，但页面应提示「该内容已有教师回答，修改或删除可能影响上下文」。第一版不做更复杂的锁定规则。

### 2.6 课程整体反馈（设计增强）

> **说明**：需求获取文档的 3.1.5「问题反馈发布模块」中，反馈与具体「作业」绑定。「课程整体反馈」不在需求获取文档范围内，但项目当前已实现对应后端 (`POST/GET/PATCH/DELETE /courses/:courseId/course-feedbacks` 等) 与数据表 `course_feedbacks(course_id, student_id, dimension, content)`。为不浪费已有能力，Web 端保留这一 Tab，但限定为与后端 schema 完全一致的最简形态：扁平的「维度 + 内容」直接提交，**不引入任务、生命周期、匿名等需求文档没有的概念**。

学生在课程工作区「课程整体反馈」Tab 中可以随时（在课程「开课中 / 已结课」时）提交课程层面的反馈。

字段（与后端 `course_feedbacks` 表对齐）：

- 反馈维度（dimension）：枚举 5 个值，与后端 CHECK 约束一致：
  - `content` 课程内容
  - `method` 教学方法
  - `teaching` 教师授课
  - `gain` 学习收获
  - `other` 其他建议
- 反馈内容（content）：纯文本。

学生可以：

- 提交课程整体反馈（多次提交，按维度区分）。
- 查看自己已提交的课程整体反馈。
- 修改 / 删除自己已提交的课程整体反馈（对应 `PATCH/DELETE /course-feedbacks/:feedbackId`）。

页面不提供匿名选项，不展示提交人脱敏字段。

## 3. 教师端

### 3.1 左侧导航

```text
工作台
授课课程
作业管理
教学任务
账号维护
```

| 入口 | 职责 |
| --- | --- |
| 工作台 | 跨课程教学待办与摘要：待批改提交、未回答作业反馈、即将截止作业、最近发布的课程整体反馈数。待办由 `submissions`、`feedbacks` 当前 API 派生。 |
| 授课课程 | 教师本人授课课程列表；进入某门课程后进入「课程工作区」。 |
| 作业管理 | 跨课程的「作业对象」中心：发布、修改、取消、查询作业，查看提交统计。 |
| 教学任务 | 教师「待办流」：待批改提交、未回答作业反馈两条任务队列。 |

`作业管理` 与 `教学任务` 必须保持对象边界清晰：

- `作业管理` 关注作业对象本身（作业的创建、维护、状态、提交统计）。
- `教学任务` 关注教师当前需要处理的事项（具体哪一条提交需要批改、哪一条反馈需要回答）。

### 3.2 工作台

教师工作台不放置复杂业务表单，只做汇总和跳转。

建议展示：

- 当前授课课程数
- 已发布作业数
- 待批改提交数
- 未回答作业反馈数
- 即将截止的作业
- 收到的课程整体反馈条数

工作台中的待办项点击后跳转到对应课程、作业、提交或反馈线程，并自动带入上下文。

### 3.3 授课课程与课程工作区

教师从「授课课程」进入某门课程后，进入课程工作区。

顶部固定展示课程上下文：

```text
课程名称 / 学期 / 课程状态 / 上课时间地点 / 学生人数 / 作业数 / 待处理数
```

Tab：

```text
课程概览 / 作业 / 提交批改 / 作业反馈 / 课程整体反馈
```

> 「课程整体反馈」Tab 与 §2.6 一致，是设计增强；教师端为只读汇总视图。

#### 3.3.1 课程概览

帮助教师快速理解当前课程运行状态。

主要内容：

- 课程简介
- 学期
- 上课时间与地点
- 课程状态
- 学生人数
- 最近作业
- 待批改提交摘要
- 未回答作业反馈摘要

教师不在该页面编辑课程基础信息；课程基础信息维护以教务员端为主。

#### 3.3.2 作业

「作业」Tab 用于管理当前课程下的作业，覆盖需求获取文档 3.1.3 的全部能力：

- 发布作业（对应 `POST /courses/:courseId/assignments`）
- 修改作业（对应 `PATCH /assignments/:id`）
- 取消作业（对应 `POST /assignments/:id/cancel`）
- 查询作业（对应 `GET /courses/:courseId/assignments`）

约束：

- 作业必须绑定课程；从课程工作区发布作业时，自动带入当前课程。
- 教师只能为自己授课的课程发布作业（由后端校验 `teacher_id` 一致）。
- 修改作业遵循需求文档约束：教师在作业截止前可以对已发布但未被学生提交的作业进行修改；UI 在不满足条件时禁用「修改」按钮并显示原因。
- **取消作业 = 作业状态置为「已取消」 + 物理清除该作业下所有提交记录**（与需求文档 3.1.3③ 一致，与后端实际行为一致）。
- 取消作业的弹窗中要求教师填写取消原因，并显式提示「此操作将清除所有学生在该作业下的提交记录，且不可恢复」，避免误操作。

#### 3.3.3 提交批改

「提交批改」Tab 用于处理学生答案，对应需求获取文档 3.1.4③「批改答案」。

教师在该页面完成：

- 查看学生提交答案（对应 `GET /assignments/:id/submissions` 与 `GET /submissions/:id`）。
- 填写分数与教师批改回复，保存批改结果（对应 `POST /submissions/:id/grade`）。

批改页面必须展示完整上下文：

```text
课程 -> 作业 -> 学生 -> 提交内容 -> 分数 / 批改回复
```

教师可以重新提交批改结果（覆盖式更新分数和批改回复），不保留批改版本历史。若该提交已经产生作业反馈，UI 应提示「修改批改结果可能影响已有沟通上下文」。

第一版不做：打回重交、批改版本历史、独立成绩分析入口。

#### 3.3.4 作业反馈

「作业反馈」Tab 用于处理学生在作业批改后提出的问题或反馈，对应需求获取文档 3.1.5 与 3.1.6。

列表展示：

- 课程名
- 作业名
- 学生姓名 / 学号
- 反馈类型（question / feedback）
- 反馈内容摘要
- 发布时间
- 回答状态：未回答 / 已回答（由是否已有 response 自然推导）
- 操作入口

反馈详情展示：

- 课程信息（只读）
- 作业信息（只读）
- 学生提交答案（只读）
- 教师批改分数与批改回复（只读）
- 学生问题或反馈
- 教师回答列表（按时间排列）
- 新增回答表单

教师对答疑的操作（对应需求获取文档 3.1.6）：

- 增加回答（`POST /feedbacks/:id/responses`）
- 查看回答（在线程内自然展示）
- 修改回答（`PATCH /responses/:id`，且仅限自己发布的回答）
- 删除回答（`DELETE /responses/:id`，且仅限自己发布的回答）

一个学生问题或反馈下允许教师发布多条回答。第一版只做按时间排列的回答列表，不做楼中楼。

第一版不做：标记已解决 / 重新打开 / 额外反馈状态流转。

#### 3.3.5 课程整体反馈（设计增强）

教师在课程工作区「课程整体反馈」Tab 中可以只读查看本课程的所有学生课程整体反馈条目，按维度分组展示（对应 `GET /course-feedbacks?courseId=...`）。

教师可看到：

- 各维度反馈数量
- 反馈明细（学生姓名、维度、内容、提交时间）

教师不在该 Tab 中编辑或删除学生提交的课程整体反馈。

### 3.4 作业管理

「作业管理」是跨课程作业中心，适合教师快速处理所有授课课程下的作业对象。

支持能力：

- 按课程筛选
- 按作业状态筛选
- 按截止时间排序
- 发布、修改、取消、查询作业（全部能力与「课程工作区→作业」Tab 一致，区别只是入口角度）
- 进入某个作业的提交批改页

从「作业管理」发布作业时，教师必须先选择课程；可选课程仅限教师本人授课课程。如果教师当前没有授课课程，发布入口应禁用，并提示联系教务员分配课程。

### 3.5 教学任务

「教学任务」是教师待办处理台，**不是作业对象管理页**。

第一版包含两类任务队列：

```text
待批改提交
未回答作业反馈
```

待批改提交任务展示：

- 课程名 / 作业名 / 学生姓名 / 学号 / 班级 / 提交时间 / 进入批改

未回答作业反馈任务展示：

- 课程名 / 作业名 / 学生姓名 / 学号 / 反馈类型 / 内容摘要 / 发布时间 / 进入回答

从任务进入详情时，系统自动带入课程、作业、提交、反馈上下文，不要求教师手动跨多个页面重新选择。

### 4.1 左侧导航

教务员工作台用于展示教学运行总览。

建议展示：

- 课程总数
- 开课中课程数
- 已结课课程数
- 教师数
- 学生数
- 最近新增课程
- 异常提醒（只读、可跳转，不自动处理）

异常提醒第一版包括：

- 课程开始日期已到但状态仍为未开课
- 课程结束日期已过但状态仍为开课中
- 被禁用账号数

点击提醒后跳转到对应页面或筛选结果。

### 4.3 课程运营

课程运营是教务员端主入口，覆盖需求获取文档 3.1.2 全部四个子功能。

课程列表支持：

- 添加课程（对应 `POST /courses`）
- 按课程名称 / 课程代码搜索
- 按授课教师筛选
- 按学期筛选
- 按课程状态筛选
- 查看课程人数、授课教师、课程状态
- 进入课程详情

课程详情 Tab：

```text
课程概览 / 基础信息维护 / 作业概况 / 课程反馈查看
```

#### 4.3.1 课程概览

只读展示课程当前运行状态：

- 课程名称、课程代码、学期、授课教师、上课时间与地点
- 课程状态
- 学生人数、作业数、课程反馈条数

#### 4.3.2 基础信息维护

对应需求获取文档 3.1.2 的「添加 / 修改 / 删除课程」。

可编辑字段（与后端 `courses` 表一致）：

- 课程名称、课程代码、授课教师、开课学期、课程描述、授课地点、上课时间（schedule_text）、课程人数上限、开始日期、结束日期、课程状态。

修改课程时按需求获取文档 3.1.2③ 要求弹出二次确认。

删除课程（对应 `DELETE /courses/:id`）：

- 按需求获取文档 3.1.2④，删除课程会同步级联删除该课程下的选课、作业、提交、批改、作业反馈、课程反馈等关联数据，对应后端 SQLite 外键 `ON DELETE CASCADE`。
- UI 必须在确认弹窗中显式列出影响范围（关联作业数、关联提交数、关联反馈数），让教务员明确知道删除是不可恢复操作。
- 文档不再使用「停止开课 / 归档」表述。如果教务员希望保留历史记录但不让课程继续开放，应通过修改课程状态字段为 `suspended` 实现，而不是走删除路径。

#### 4.3.3 作业概况

只读查看课程作业运行情况，**不替代教师端作业管理**。

展示：

- 作业列表
- 作业状态
- 发布时间、截止时间
- 提交人数、待批改人数、已批改人数

教务员不在此页面发布、修改、取消、批改作业。

#### 4.3.4 课程反馈查看（设计增强）

只读查看该课程下学生提交的课程整体反馈明细：

- 学生姓名、学号
- 维度
- 反馈内容
- 提交时间

教务员在第一版不编辑、不删除学生反馈。

### 4.4 用户管理

用户管理按账号类型分 Tab：

```text
学生账号 / 教师账号 / 教务员账号
```

#### 4.4.1 学生账号

学生账号由学生自助注册（对应 `POST /auth/register/student`）。

教务员可以：

- 查看学生列表（对应 `GET /users?role=student`）
- 按姓名 / 学号 / 手机号搜索
- 按账号状态筛选
- 查看学生详情
- 禁用账号 / 恢复账号（对应 `PATCH /users/:userId/status`）

教务员不直接修改学生个人资料，不重置学生密码。

#### 4.4.2 教师账号

按需求获取文档 3.1.1①「教师和教务员无需注册，由账号由后台分配」，教师账号由系统管理员在初始化阶段通过 seed 预置。教务员端**不提供创建教师账号的入口**。

教务员可以：

- 查看教师列表（对应 `GET /users?role=teacher`）
- 按姓名 / 工号 / 手机号搜索
- 按账号状态筛选
- 查看教师详情
- 禁用账号 / 恢复账号（对应 `PATCH /users/:userId/status`）

教师忘记密码时走手机号验证码找回流程（对应 `POST /auth/password/forgot`），教务员端不提供重置教师密码入口。

#### 4.4.3 教务员账号

教务员账号由系统管理员通过 seed 预置。

普通教务员端规则：

- 可以只读查看教务员账号列表（对应 `GET /users?role=officer`）。
- 不创建教务员账号。
- 不禁用其他教务员账号。
- 不重置教务员密码。

如需新增、禁用或恢复教务员账号，应由系统管理员在更高权限后台或运维流程中处理（demo 项目中通过修改 seed 实现）。

### 4.5 课程反馈查看（全局）

- 课程 / 授课教师 / 学期
- 反馈维度
- 反馈内容
- 提交时间
- 提交人（学生姓名 / 学号）

教务员第一版不编辑、不删除学生反馈。

### 4.6 账号维护

只处理当前教务员自己的账号信息：

- 查看个人资料（只读）
- 修改手机号
- 修改密码
- 注销账号 / 退出登录

## 6. 一致性结论

- 需求获取文档 3.1.1 ~ 3.1.6 共 25 个功能点全部由现有后端 API 支持。

## 7. 给后续前端实现会话的工作提示

> 本节针对在新会话中按本文档实施前端重构的开发者 / Claude。开工前请先完整阅读本文档（特别是 §1.1 / §1.2 / §1.3 / §5），再动 `apps/web/`。

### 7.1 基本原则

- **本文档是设计依据**。需求获取报告里没有的能力——尤其是「匿名提交」「消息中心」「课程反馈任务制生命周期」「打回重交」「批改版本历史」「标记已解决」——不要再加回来。这些都是评审里已经被否决的扩张。
- **不改后端 / 不改 schema**。如果实现过程中发现某个需求需要后端调整，先在 §5 对齐表里找对应接口；找不到再回头与项目负责人确认，不要自行新增 API 或修改 `apps/server/src/lib/db/schema.ts`。
- **不引入新的全局状态库**。`apps/web` 当前依赖 TanStack Query + 路由参数 + 局部 useState 做状态管理，不要新增 Redux / Zustand / Recoil 等。
- **保留课设交付物的可追溯性**。本系统是软件工程课程设计的最终交付，验收依据是 `doc/` 下的需求获取与需求分析报告。所有设计偏离都已在 §5 / §6 显式标注，新会话不再扩张。

### 7.2 仓库约定

- 本项目无路径别名，跨包导入使用长相对路径（如 `../../../../../packages/shared/src/index`）。不要新增 tsconfig `paths`。
- 后端没有 ORM，所有业务规则都在 Fastify 路由里。重构前端时如果需要确认某个业务规则（比如「作业反馈是否要求 submission 已批改」），直接读 `apps/server/src/modules/` 下对应路由文件。
- 共享 zod schema 在 `packages/shared/src/index.ts`，前端只能消费不能修改其结构。
- React/ReactDOM 通过 `apps/web/vite.config.ts` 的 `createRequire` 解析做了去重，不要随意改这段配置。
- Vite 端口默认 5173 会被 `scripts/dev.sh` 自动让位；端口浮动由 `scripts/lib/runtime-info.sh` 解析，不要在前端代码里硬编码端口。
