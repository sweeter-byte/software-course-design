# Web 与 Mobile 人工验收演示检查清单

生成日期：2026-05-04

本文档用于结课前人工验收演示检查。检查目标是确认 Web 端、Mobile 端、后端接口和数据库使用同一套真实数据；同时覆盖常见易出错问题，避免答辩现场只演示“顺手路径”。

### 1.5 截图保存与命名规则

建议把所有验收截图统一保存到：

```text

```


截图要求：

1. 截图必须能看出当前端是 Web 还是 Mobile。
2. 截图尽量包含当前角色身份，例如教师、学生、教务员。
3. 截图必须包含关键业务数据，例如课程代码 `ACC-0504`、作业标题、反馈内容或分数。
4. 不要专门截密码明文；注册、登录、改密类截图重点截成功提示或登录后的身份页面。
5. 如果同一步 Web 和 Mobile 都要证明同步，必须各截一张，文件名连续编号。

### 1.6 是否需要重置数据库

如果是正式答辩，不建议临场重置数据库。  
如果只是自查，希望从干净状态开始，可以执行：

```bash
npm run db:reset --workspace @course/server
```

注意：重置数据库会删除已有学生、课程、作业、提交和反馈，只保留种子教师/教务员账号。

## 2. 基础连通检查

### 2.1 后端健康检查

1. 打开 `http://localhost:4100/health`。
2. 预期看到健康状态响应，例如 `healthy`。

建议截图：

| 截图内容 | 文件名 |
| --- | --- |
| 浏览器打开健康检查接口并显示 healthy | `01-api-health.png` |

易错点：

| 问题 | 现象 | 处理 |
| --- | --- | --- |
| 后端没启动 | Web/Mobile 登录时报网络错误 | 先启动 `npm run dev` 或 `npm run dev --workspace @course/server` |
| 端口被占用 | 后端启动失败 | 关闭旧进程后重启 |
| 数据库状态混乱 | 数据与预期不一致 | 自查阶段可执行 `db:reset`，正式答辩前慎用 |

### 2.2 Web 端加载检查

1. 打开 Web 地址，例如 `http://localhost:5173/`。
2. 确认显示课程互动管理系统登录页。
3. 确认默认只显示“账号登录”表单。
4. 确认“忘记密码？”和“学生注册”位于登录框下方，找回密码表单不会默认展开。

建议截图：

| 截图内容 | 文件名 |
| --- | --- |
| Web 默认登录页，只显示账号登录和下方入口 | `02-web-login-default.png` |

易错点：

| 问题 | 现象 | 处理 |
| --- | --- | --- |
| 打开了 API 地址 | 页面不是 Web 登录页 | 打开 Web 地址，不要打开 `4100` API 地址 |
| 页面旧样式未更新 | 登录页仍显示旧布局 | 强制刷新浏览器，或确认启动的是当前项目目录 |

### 2.3 Mobile 端加载检查

1. Android 模拟器启动后，执行：

```bash
npm run android --workspace @course/mobile
```

2. Mobile 页面打开后，在 `API Base URL` 输入框填：

```text
http://10.0.2.2:4100/api/v1
```

3. 确认顶部显示“课程互动 · 移动工作台”，登录区默认只显示账号登录。

建议截图：

| 截图内容 | 文件名 |
| --- | --- |
| Mobile 默认登录页，API 地址为 `10.0.2.2`，只显示账号登录 | `03-mobile-login-default.png` |

易错点：

| 问题 | 现象 | 处理 |
| --- | --- | --- |
| Mobile 填了 `localhost` | 模拟器登录失败或网络错误 | 改为 `http://10.0.2.2:4100/api/v1` |
| 模拟器未连接 | Expo 无法打开 App | 先在 Android Studio Device Manager 启动模拟器，再执行 `adb devices` 检查 |
| 端口没转通 | Mobile 请求失败但 Web 正常 | 确认后端监听 `0.0.0.0:4100`，并使用 `10.0.2.2` |

## 11. 常见问题快速排查

| 问题 | 判断方式 | 快速处理 |
| --- | --- | --- |
| Web 能用但 Mobile 不能登录 | Web 登录正常，Mobile 报网络错误 | Mobile API Base URL 改为 `http://10.0.2.2:4100/api/v1` |
| Web 和 Mobile 数据不同步 | 一端新增后另一端看不到 | 另一端先刷新；Mobile 可在 Expo 终端按 `r` Reload |
| 教师看不到课程 | 教师课程列表为空 | 检查课程的授课教师编号是否为 `teacher-demo-001` |
| 学生看不到作业 | 学生未加入课程或未选课程 | 先加入课程，再选择课程 |
| 提交/反馈按钮不可用 | 未选中课程、作业或提交 | 按“课程 -> 作业 -> 提交”的顺序依次选中 |
| 验证码失败 | 验证码用途不匹配 | 注册、找回密码、改手机号分别获取对应验证码 |
| 手机号重复 | 注册或改号失败 | 换一个未使用手机号 |
| 删除/注销后数据没了 | 执行了破坏性操作 | 破坏性操作只对临时课程、临时作业、专项测试账号执行 |
| 刷新后学生端“查看提交”不可用 | 当前提交选择状态丢失 | 提交成功后不要立即刷新；如已刷新，重新走一份新作业提交演示 |

## 12. 验收完成判定

人工验收可判定通过的最低标准：

1. Web 和 Mobile 都能登录教师、教务员、学生账号。
2. 学生注册、登录、找回密码至少能在一个端完整跑通。
3. 教务员能创建、修改、删除课程。
4. 教师能发布、修改、取消作业。
5. 学生能加入课程、提交答案、修改答案。
6. 教师能查看提交并批改。
7. 学生能提交课程反馈、修改和删除自己的反馈。
8. 学生能围绕作业提交发布问题/反馈，教师能回复。
9. Web 写入的数据，Mobile 刷新后能看到。
10. Mobile 写入的数据，Web 刷新后能看到。
11. 常见错误路径有明确提示或能被正确拦截，例如重复手机号、错误密码、重复选课、未选课程就操作、验证码用途错误。

建议验收时每完成一个主步骤就截图或录屏保留证据，后续写《系统测试计划》和《结项报告》时可直接引用。最终截图文件建议按编号放入 `docs/acceptance-screenshots/`，并在报告中引用截图编号，例如“见图 13 Web 创建课程成功”和“见图 14 Mobile 同步显示课程”。
