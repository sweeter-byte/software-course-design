# 课程互动管理系统项目概要

## 1. 项目背景

依据 `doc/001 B 课程互动管理系统 开题报告.docx`、`doc/001 B 课程互动管理系统 系统设计报告.docx`、`doc/001 B 课程互动管理系统 需求分析报告.docx`、`doc/001 B 课程互动管理系统 需求获取报告 5.docx`，本系统面向高校课程教学场景，目标是把分散的课程通知、作业、答疑和反馈流程统一到一个可持续迭代的平台中，同时提供 Web 端和移动端，共享同一后端业务语义与数据源。

## 2. 目标用户

- 学生：注册登录、维护个人信息、查看课程、提交作业、查看批改结果、发布问题或反馈。
- 教师：登录、维护个人信息、发布作业、批改答案、查看学生反馈、进行答疑。
- 教务员：登录、维护个人信息、维护课程信息、统筹账号与课程运行情况、处理异常。

## 3. 核心业务目标

- 提供统一的课程信息入口，减少信息分散与遗漏。
- 支持作业发布、提交、批改、反馈的完整闭环。
- 支持围绕作业结果的提问与反馈沉淀，形成课程互动记录。
- 让 Web 与移动端共用同一套 API、数据结构和权限语义。
- 通过日志、测试、文档和演示数据，交付一个课程设计/答辩可用的完整工程。

## 4. 文档提炼出的核心模块

### 4.1 用户与认证

- 学生手机号注册
- 三角色登录
- 个人信息修改
- 密码修改与找回
- 账号注销

### 4.2 课程管理

- 教务员创建、查询、修改、删除课程
- 多条件课程检索
- 课程详情展示
- 学生查看个人已选课程

### 4.3 作业管理

- 教师发布作业
- 教师修改作业
- 教师取消作业并填写原因
- 教师/学生按课程查看作业

### 4.4 作业提交与批改

- 学生提交答案
- 学生在截止前或未批改前修改答案
- 教师批改并给出分数与评语
- 学生查看提交状态与批改结果

### 4.5 问题/反馈与答疑

- 学生围绕作业批改结果发布问题或反馈
- 学生查看、修改、删除自己的问题/反馈
- 教师查看问题/反馈并回复
- 教师查看、修改、删除自己的回复

### 4.6 运维与质量

- 统一日志
- 权限控制
- 数据一致性与恢复策略
- 测试计划、测试报告、部署说明

## 5. 关键非功能需求

- 常见读写操作响应目标在 1 秒左右，整体服务响应不超过 1.5 秒。
- 用户隐私、口令安全、角色权限和输入校验必须可说明、可实现。
- 数据记录量目标约 1 万级，结构化持久化存储。
- Web、移动端和后端必须围绕单一事实源运行。
- 项目必须具备清晰的模块边界、文档说明和可演示性。

## 6. 当前设计结论

- 构建 TypeScript monorepo：`apps/web`、`apps/mobile`、`apps/server` + `packages/*`。
- 以后端为唯一事实源，Web 和移动端通过共享 DTO、校验模型和 REST API 同步。
- 采用关系型数据建模，保留向 MySQL 迁移的边界；本地演示优先使用轻量 SQL 存储。
- 优先完整交付认证、课程、作业、提交、反馈/答疑五条核心业务链路。
