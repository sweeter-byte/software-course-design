# 课程互动管理系统设计说明

## 1. 设计目标

围绕课程文档中明确的五条核心业务链路交付一个可运行、可演示、可答辩的系统：

- 认证与账号治理
- 课程管理
- 作业发布与批改
- 学生提交与结果查看
- 问题/反馈与答疑

同时满足 Web 与移动端共享同一套业务真相、统一日志、统一权限和可追踪测试。

## 2. 方案比较

### 方案 A：按文档原型复刻 `JSP + Spring Boot + Android Native`

- 优点：与原文档技术描述最接近。
- 缺点：同时交付 Web、移动端和共享契约的成本高，前后端复用差，短周期内难做到高质量 UI 与一致性。

### 方案 B：TypeScript Monorepo（推荐）

- Web：React/Vite
- Mobile：Expo React Native
- Server：Fastify/TypeScript
- Shared：共享 DTO、校验模型、常量和 API client

优点：

- Web 与移动端共享类型和数据语义，最容易满足同步要求。
- 前后端都适合模块化与测试。
- UI 质量、开发效率与答辩可解释性平衡最好。

缺点：

- 与原始 Java/JSP 描述存在实现栈差异，需要在假设文档中说明。

### 方案 C：Python Backend + React/Expo

- 优点：后端开发效率高，测试便利。
- 缺点：共享类型与前后端契约需要额外生成流程，仓库一致性不如全 TypeScript。

最终选择：方案 B。

## 4. 关键设计决策

### 4.1 反馈模型

把文档里的“问题”和“反馈”收敛为同一实体，使用 `kind` 区分，避免重复流程与重复表结构。

### 4.2 同步机制

不强行引入高成本实时链路，采用统一 API + 写后失效刷新 + 定时轮询 + 手动刷新，保证演示中的一致性。

### 4.3 日志机制

后端所有关键请求和领域动作落盘到 `logs/`，并在数据库中保留审计事件，便于排错与答辩说明。

## 5. 错误处理

- 表单校验错误：字段级提示
- 权限错误：统一 403 响应
- 业务冲突：例如重复选课、截止后提交、已批改后修改
- 系统异常：统一错误码 + 请求 ID + 错误日志

## 6. 测试策略

- 领域服务单元测试
- 认证、课程、作业、提交、反馈 API 集成测试
- 关键流程手工测试清单
- 构建、类型检查、Lint 校验

## 7. 交付边界

本次优先确保核心业务闭环完整，附件上传、真实短信网关、推送通知和高级统计图保留扩展接口，但不作为首版阻塞项。
